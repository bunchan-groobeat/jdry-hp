この提案書は、この「専用フォーム → 自動反映」体験をどれだけ忠実に実現できるかを最優先の評価軸として、3案を比較します。サイトは **GitHub Pages の静的サイトのまま維持**することが前提です。

## 案1: Decap CMS（旧 Netlify CMS）〔専用管理画面 → GitHubに直接コミット〕

### 仕組みの概要
サイトに `/admin` という**専用の管理画面**を1枚追加します。J-DRYさんはスマホのブラウザでそこを開き、ログイン後、記事フォーム（タイトル・本文・写真）に入力して「Publish」。Decap CMS が記事（Markdown）と画像を **GitHubリポジトリに直接コミット** → GitHub Pages が再ビルドして反映。

### 理想像の充足度
- **専用の入力画面**: ◎ タイトル欄・本文欄・画像アップロード欄がそろった専用フォームが用意できる。
- **投稿→自動反映**: ○ コミット後にGitHub Pagesのビルドが走り、**反映まで概ね1〜3分**。
- **写真アップ**: ○ 画面から写真を選べるが、**自動リサイズ機能は標準では弱い**（大きいスマホ写真がそのままリポジトリに入る）。別途、ビルド時に画像を圧縮する処理を組めば解消可能。

### J-DRYさんの投稿手順
1. スマホで `https://（サイト）/admin` を開く
2. GitHubでログイン（初回のみ認証）
3. 「新規」→ タイトルを入力
4. 本文を入力（見たままに近い簡易エディタ）
5. 写真を選んでアップロード
6. 「Publish」を押す → 1〜3分でTopicsに反映

### メリット
- **完全無料**（GitHub＋Cloudflare Workers 無料枠）。
- 記事も画像も**自分のGitHubリポジトリに残る**（データが手元にある／ロックインが弱い）。

### デメリット
- ログイン認証の設定が手間。Netlifyを使わない場合、GitHub OAuth の中継（Cloudflare Workers 等・無料）を1つ立てる必要がある。
- 記事をHTML化するビルド（GitHub Actions＋静的サイトジェネレータ）を新設するため、構成が一段複雑になる。
- 写真の自動最適化は作り込みが必要。

### 費用 / 保守 / リスク
- 費用: **0円**。
- 保守（ジロー）: 初期構築は中〜大、以降は軽い。
- リスク: Decap CMS はOSSで開発が緩やか。認証中継を自前で持つ必要がある。

## 案2: microCMS（国産ヘッドレスCMS）〔専用管理画面で投稿 → 自動ビルドで反映〕 ★おすすめ

### 仕組みの概要
**microCMS**（日本製のヘッドレスCMS）に「お知らせ」入力フォーマットを作成。J-DRYさんは microCMS の**管理画面（スマホ対応・完全日本語）**で、タイトル・本文・写真を入力して「公開」。公開と同時に Webhook が発火 → **GitHub Actions が起動 → 記事を取得してTopicsページを再生成 → GitHub Pages に反映**。サイトは静的のまま。

### 理想像の充足度
- **専用の入力画面**: ◎ まさに「タイトル欄・本文欄・画像欄」がそろった専用フォーマットを自由に設計できる。社長の理想像そのもの。
- **投稿→自動反映**: ◎ 公開ボタンでWebhook即発火、**反映まで概ね1〜2分**。下書き・公開予約・プレビューも可能。
- **写真アップ**: ◎ **スマホの写真をそのままドラッグ＆ドロップでアップOK**。microCMSが画像を配信最適化（サイズ/フォーマット変換をURLパラメータで自動化）でき、リサイズを気にせず投稿できる。

### J-DRYさんの投稿手順
1. スマホで microCMS にログイン（ブックマークから）
2. 「お知らせ」→「新規作成」
3. タイトルを入力
4. 本文を入力（リッチな日本語エディタ、箇条書き・リンクも簡単）
5. 写真をアップ（スマホのカメラロールから選ぶだけ、リサイズ不要）
6. 「公開」を押す → 1〜2分でTopicsに反映

### メリット
- **管理画面が最も分かりやすく完全日本語**。非エンジニアのJ-DRYさんの学習コストが最小。
- 記事データはmicroCMS側で管理されるので**リポジトリが重くならない**。
- ブログらしい機能（下書き・予約・履歴）が最初からそろう。

### デメリット
- 記事データが microCMS 側に保存される（SaaS依存）。将来やめる場合はエクスポートが必要。
- 無料プランに上限あり（後述）。お知らせ用途なら十分だが、大規模化時は課金可能性。
- GitHub Actions のビルド設定が必要（案1同様、静的サイトジェネレータを1本挟む）。

### 費用 / 保守 / リスク
- 費用: **無料プラン（Hobby）0円**。API・メンバー数などに制限があるが、お知らせ数十〜数百件・1人運用なら収まる想定。拡大時は Team プラン 月額 **約4,900円〜**（当面不要）。
- 保守（ジロー）: 初期のスキーマ設計＋Webhook＋Actionsビルド構築に工数。以降は軽い。
- リスク: 外部SaaS依存。障害時は「新規公開が一時できない」だけで、既存サイトは静的なので**表示は継続**。

## 案3: Google フォーム 投稿 → 半自動反映〔いちばん手軽だが管理に弱い〕

### 仕組みの概要
J-DRYさんは使い慣れた **Googleフォーム**に「タイトル・本文・写真」を入力して送信。回答が Google スプレッドシートに溜まる → GitHub Actions（定期実行 or 手動）がそれを読み、Topicsを再生成。

### 理想像の充足度
- **専用の入力画面**: ○ フォーム自体は「タイトル・本文・写真」の専用画面として成立するが、**入力専用で「管理画面」ではない**（あとから直せない）。
- **投稿→自動反映**: △ フォーム送信からサイト反映まで、画像の取り回しなどで**ジローの手動ケアや定期ビルド待ちが入りやすく、"数分以内に自動" を安定保証しにくい**。
- **写真アップ**: △ フォームからのアップは可能だが Google ドライブ保存になり、**サイト用への変換・リサイズに作り込みが必要**。

### J-DRYさんの投稿手順
1. スマホでフォームのURLを開く（ブックマーク）
2. タイトルを入力
3. 本文を入力
4. 写真を添付
5. 「送信」→（変換処理を経て）Topicsに反映

### メリット / デメリット
- メリット: 投稿手順が**世界一かんたん**・新ツール不要。フォームは無料。
- デメリット: **編集・削除・並べ替えがしにくい**。誤字修正のたびにスプレッドシート操作。ちゃんとしたブログに育てるには結局案1/案2へ作り直しになりやすい。

### 費用 / 保守 / リスク
- 費用: **0円**。
- 保守（ジロー）: 変換ロジック構築＋**日常的な軽い手動ケアが残る**。
- リスク: 「投稿はできるが管理は不便」で、運用が育つと破綻しやすい。つなぎ／簡易版向き。

## ジローのおすすめ案と理由

### おすすめ: **案2 microCMS ＋ GitHub Actions 自動ビルド**

社長の理想像「専用フォーマットに打ち込んで投稿 → 数分で自動反映、写真もそのまま」に**最も忠実**で、費用・保守も現実的だからです。

1. **理想の投稿体験にそのまま合致**。タイトル・本文・写真の専用フォーマットを自由に作れ、公開ボタンで**1〜2分で自動反映**。写真は**スマホからそのままアップ、リサイズも自動**——社長の言葉どおりの体験になる。
2. **管理画面が完全日本語で最も分かりやすい**。J-DRYさん（非エンジニア）が迷わず使え、HTMLもGitも触らない。
3. **無料枠で開始でき**、お知らせ用途なら当面課金不要。将来はプラン変更で素直にスケール。
4. サイトは**静的のまま**（GitHub Pages維持）で表示は高速・安定。SaaS障害時も既存ページは生き続ける。
5. ジロー側の日常保守が軽い（案3のような手動ケアが残らない）。

### 補足・進め方
- 「記事データを自分のGitに全部残したい」を最重視するなら **案1 Decap CMS**。思想的にはロックインが最も弱い。ただし写真の自動リサイズと認証周りに一手間。
- 「まず最速で投稿だけ始めたい」なら **案3を暫定運用** → 案2へ移行、も可。ただし遠回りになりがち。
- いずれも、まず**ジローが試験投稿1件を作って反映までデモ**し、社長に実際にスマホで触ってもらってから本採用を決めるのが安全。
